# spec-driven-development

A self-replicating agentic workflow factory that treats **agent definitions as software that must be compiled** for specific project contexts.

**Key Concept**: Install the plugin once globally (`~/.claude/`), then compile custom agents for each project (`./.claude/`).

## Two-Tier Architecture

| Tier | Location | Installed From | Purpose | Frequency |
|------|----------|----------------|---------|-----------|
| **Global Factory** | `~/.claude/` | `plugins/spec-driven-development/` | Reusable agent creation tools | Once per machine |
| **Local Products** | `./.claude/` | Generated by skills | Project-specific agents | Once per project |

**The Factory** (14 skills + 1 meta-agent):

Project compilation:
- `/compiling-agentic-workflow` - **One-command full setup**: CLAUDE.md + docs/ + planner + builder
- `/compiling-project-settings` - Generate lean CLAUDE.md (50-100 lines)
- `/compiling-project-docs` - Generate `docs/` for progressive disclosure

Agent compilers:
- `/compiling-planner-agent` - Compile project-specific planner
- `/compiling-builder-agent` - Compile project-specific builder
- `/compiling-security-agent` - Compile security auditor (red team)
- `/compiling-devops-agent` - Compile DevOps/infrastructure specialist

Specification skills:
- `/defining-specs` - Turn PRDs/briefs into behavioral specs
- `/defining-test-scenarios` - Expand specs into exhaustive test scenarios

Orchestration:
- `/orchestrating-workflow` - Decompose features, plan in parallel, build in parallel

Utility skills:
- `/reviewing-code-quality` - Score code against 6 quality axes
- `/stress-testing` - Adversarial property-based tests
- `/git-commit-changes` - Split git changes into atomic commits
- `/conducting-post-mortem` - Extract lessons, propose CLAUDE.md updates

Meta-agent:
- `agent-author` - Expert guide for creating custom agents and skills

**The Products (compiled per-project):**
- `CLAUDE.md` - Lean project context (50-100 lines)
- `docs/` - Detailed documentation (architecture, workflow, tech-stack, getting-started)
- `.claude/agents/planner.md` - Planner specialized for THIS tech stack
- `.claude/agents/builder.md` - Builder specialized for THIS codebase
- `.claude/agents/security.md` - (optional) Security auditor
- `.claude/agents/devops.md` - (optional) DevOps specialist

## Step 1: Install the Global Factory (One-Time)

### Unix/Mac/Linux

```bash
cd /path/to/pipemind-marketplace

# Symlink (recommended — git pull auto-updates)
./install.sh spec-driven-development --symlink

# Copy method
./install.sh spec-driven-development

# Verify
ls ~/.claude/agents/         # Should show: agent-author.md
ls ~/.claude/skills/         # Should show: 14 skill directories
```

### Windows (PowerShell)

```powershell
cd C:\path\to\pipemind-marketplace

Copy-Item -Recurse plugins\spec-driven-development\agents $env:USERPROFILE\.claude\
Copy-Item -Recurse plugins\spec-driven-development\skills $env:USERPROFILE\.claude\

dir $env:USERPROFILE\.claude\agents\
dir $env:USERPROFILE\.claude\skills\
```

## Step 2: Compile Agents for a Project

**Prerequisites:**
- Global factory installed (Step 1)
- [Claude Code CLI](https://github.com/anthropics/claude-code) installed
- Git repository initialized in your project

### The Compilation Sequence

Navigate to your project and run one command:

```
/compiling-agentic-workflow    → CLAUDE.md + docs/ + planner + builder (all at once)
```

This orchestrates the full sequence automatically. Or invoke each step individually:

```
/compiling-project-settings     → Lean CLAUDE.md (project context, 50-100 lines)
        ↓
/compiling-project-docs        → Detailed docs/ (architecture, workflow, tech-stack, getting-started)
        ↓
/compiling-planner-agent       → .claude/agents/planner.md (task planning)
        ↓
/compiling-builder-agent       → .claude/agents/builder.md (implementation)
        ↓
Ready for agentic workflow!
```

**Optional specialists** (add after the core workflow):
```
/compiling-security-agent      → .claude/agents/security.md (red team auditor)
/compiling-devops-agent        → .claude/agents/devops.md (infrastructure SRE)
```

**Why this order matters:**
1. `CLAUDE.md` establishes project context that all agents reference
2. `docs/` provides the detailed knowledge agents need via progressive disclosure — without it, agent compilers either reference files that don't exist or inline details that bloat agent files past the 100-line limit
3. Planner and builder are compiled last because they reference both `CLAUDE.md` and `docs/`

### What Each Step Does

**`/compiling-agentic-workflow`** — Runs the full compilation sequence below in one invocation. Idempotent — safe to re-run when your project evolves.

**`/compiling-project-settings`** — Analyzes your codebase (tech stack, directory structure, config files, conventions) and generates a lean `CLAUDE.md` following the 80% rule: only instructions relevant to 80%+ of sessions.

**`/compiling-project-docs`** — Generates detailed documentation that agents read on-demand:
- `docs/index.md` - Documentation overview and reading order
- `docs/architecture.md` - Two-tier system design, component breakdown, data flow
- `docs/getting-started.md` - Installation, verification, common issues
- `docs/workflow.md` - Planner/builder pattern, skill invocation, task file structure
- `docs/tech-stack.md` - YAML frontmatter format, model selection, tool tiers, size limits

**`/compiling-planner-agent`** — Reads `CLAUDE.md` and `docs/`, detects your tech stack, and generates a planner agent specialized for your project's task management workflow.

**`/compiling-builder-agent`** — Reads `CLAUDE.md` and `docs/`, detects your tech stack, and generates a builder agent with language-specific patterns, testing frameworks, and coding standards baked in.

### Result

```
your-project/
├── CLAUDE.md                      # Lean project context (50-100 lines)
├── docs/
│   ├── index.md                   # Docs overview and reading order
│   ├── architecture.md            # Two-tier system design and data flow
│   ├── getting-started.md         # Installation and verification
│   ├── workflow.md                # Planner/builder pattern and task files
│   └── tech-stack.md              # YAML format, models, tool tiers
└── .claude/
    └── agents/
        ├── planner.md             # Compiled for YOUR tech stack
        ├── builder.md             # Compiled for YOUR codebase
        ├── security.md            # (optional) Security auditor
        └── devops.md              # (optional) DevOps specialist
```

## The "Compilation" Philosophy

### Why Compile Agents?

A generic agent that works for all tech stacks must be slow (asks many questions), confused (uses wrong tools), and error-prone (generates invalid code).

Compiled agents "know" your project:

| Project Type | Builder Knows | Planner Knows |
|--------------|---------------|---------------|
| **Python + FastAPI** | `pytest`, `pip`, `uvicorn`, `pydantic` | API endpoint patterns, `requirements.txt` |
| **Node + Express** | `jest`, `npm`, `nodemon`, `zod` | Route structure, `package.json` |
| **Rust + Tokio** | `cargo test`, `clippy`, `async/await` | Module system, `Cargo.toml` |

### The 150-Instruction Constraint

Claude reliably follows ~150 instructions. Claude Code's system prompt uses ~50, leaving **~100 for CLAUDE.md + all agents combined**. If you stuff too much in, Claude ignores it.

This constraint drives the entire architecture:
- **CLAUDE.md** stays lean (50-100 lines) — universal project context only
- **`docs/`** holds detailed knowledge — agents read only what's relevant per task
- **Agent files** stay lean (50-100 lines) — mission-specific instructions with pointers to `CLAUDE.md` and `docs/`
- **The 80% rule** — only include instructions relevant to 80%+ of sessions

### Re-Compiling When Stack Changes

If your project evolves (e.g., Jest to Vitest, Express to Fastify):

1. Update `CLAUDE.md` with the changes
2. Re-run `/compiling-project-docs` to update `docs/`
3. Re-run `/compiling-builder-agent` (and `/compiling-planner-agent` if architecture changed)

The skills read your current context and regenerate agents accordingly.

## Usage

**Interactive mode recommended**: Run agents interactively rather than one-shot (`-p`) mode. Interactive sessions let agents ask clarifying questions and let you monitor progress.

### Planning a Feature

```bash
claude --agent planner
```

Then in the interactive session:
```
> Create a task to add user authentication
```

The planner reads `CLAUDE.md`, explores the codebase, may ask clarifying questions, designs a solution, and writes a task file with exact file paths and implementation steps.

### Building the Feature

```bash
claude --agent builder
```

Then in the interactive session:
```
> Implement task 001
```

The builder reads the task file, implements code, writes tests, runs them, invokes `/reviewing-code-quality` as a quality gate, and reports completion status.

### Security Audit (Optional)

```bash
claude --agent security
```

```
> Audit the authentication system
```

The security agent thinks adversarially — it tries to break what the builder creates. It runs automated scanning tools, tests attack scenarios (BOLA, SQL injection, JWT tampering, XSS), and generates findings with severity levels and remediation steps.

### Infrastructure Optimization (Optional)

```bash
claude --agent devops
```

```
> Optimize the Dockerfile for production
```

The DevOps agent focuses exclusively on infrastructure configuration — never application code. It analyzes Docker, Terraform, Kubernetes, CI/CD configs for security issues, performance bottlenecks, and cost optimization.

### Adversarial Verification

```bash
claude -p "/stress-testing src/auth/login.ts"
```

Generates property-based tests with 100+ random inputs using `hypothesis` (Python), `fast-check` (JS), or `proptest` (Rust) to find edge cases the builder missed.

### Post-Mortem Learning

```bash
claude -p "/conducting-post-mortem tasks/completed/001-add-user-authentication.md"
```

Analyzes what went well and what didn't, then proposes updates to `CLAUDE.md` — your agents get smarter over time.

### Complete Development Lifecycle

```
Planner  →  Builder  →  Security  →  DevOps
(Design)    (Build)     (Audit)      (Deploy)
```

## Agent Separation of Concerns

**Planner Agent** (`claude --agent planner`):
- Thinks: Analyzes problems, designs solutions
- Plans: Creates detailed task files
- Goal: Builder can execute mechanically without design decisions
- Use interactively: Planner may ask clarifying questions

**Builder Agent** (`claude --agent builder`):
- Executes: Implements according to task file
- Tests: Writes and runs tests proactively
- Documents: Records what was actually built
- Goal: Mechanical execution, no design decisions

**Security Agent** (`claude --agent security`):
- Audits: Adversarial red-team mindset
- Stack-aware: Knows your frameworks' common vulnerabilities
- Goal: Find vulnerabilities before attackers do

**DevOps Agent** (`claude --agent devops`):
- Infrastructure only: Never touches application code
- Multi-cloud: AWS, GCP, Azure patterns
- Goal: Reliable, secure, cost-optimized infrastructure

## Troubleshooting

### Global Factory Issues

**`ls ~/.claude/agents/` shows nothing**

Verify and re-install:
```bash
# Unix/Mac
ls -la ~/.claude/agents ~/.claude/skills

# Re-install if missing
cp -rv plugins/spec-driven-development/agents ~/.claude/
cp -rv plugins/spec-driven-development/skills ~/.claude/
```

**"Command not found" for a skill**

Skills must have correct directory structure:
```
~/.claude/skills/
├── compiling-planner-agent/
│   └── SKILL.md              # Must be named SKILL.md
└── ...
```

### Compilation Issues

**Skill says "CLAUDE.md not found"**

Run `/compiling-agentic-workflow` which handles the correct order automatically. Or follow the sequence manually:
1. `/compiling-project-settings` first (creates `CLAUDE.md`)
2. `/compiling-project-docs` second (creates `docs/`)
3. Agent compilers last (they reference both)

**Agents reference docs/ files that don't exist**

You skipped `/compiling-project-docs`. Run it before compiling agents.

**"Not in a git repository"**

Initialize git first: `git init`

### Runtime Issues

**Builder uses wrong testing framework**

Update `CLAUDE.md` with the correct framework, re-run `/compiling-project-docs`, then re-run `/compiling-builder-agent`.

**Planner doesn't understand architecture**

Check that `CLAUDE.md` and `docs/architecture.md` accurately describe your project. Re-run `/compiling-project-settings` and `/compiling-project-docs` if needed.

### Verification Commands

```bash
# Check global factory
ls ~/.claude/agents/agent-author.md
ls ~/.claude/skills/*/SKILL.md | wc -l    # Should show 14

# Check project compilation
ls CLAUDE.md
ls docs/
ls .claude/agents/planner.md
ls .claude/agents/builder.md

# Test agents
claude --agent planner    # Then: "List available tasks"
claude --agent builder    # Then: "Show me the current tech stack"
```

## Customization

### Customizing the Global Factory

Edit skills in `plugins/spec-driven-development/skills/` to change how agents are generated for ALL future projects.

If using symlinks, changes are live immediately. If using copies, re-install to `~/.claude/`:
```bash
cp -r plugins/spec-driven-development/skills ~/.claude/
```

Then re-compile agents in existing projects by re-running the relevant skills.

### Customizing a Specific Project

Edit compiled agents directly — changes affect only THIS project:
```bash
vim CLAUDE.md                    # Project context
vim docs/architecture.md         # Detailed architecture
vim .claude/agents/planner.md    # This project's planner
vim .claude/agents/builder.md    # This project's builder
```


```
├── agents/agent-author.md

    └── .claude/agents/
```

## Philosophy

1. **Separation of Concerns**: Planner thinks, builder executes, security audits, devops deploys
2. **The 150-Instruction Budget**: Less is more — lean files with progressive disclosure via `docs/`
3. **The 80% Rule**: Only include instructions relevant to 80%+ of sessions
4. **Reference, Don't Duplicate**: CLAUDE.md is the single source of truth; agents point to it
5. **Compiled, Not Generic**: Agents baked with project-specific knowledge outperform generic ones
6. **Adversarial Testing**: Security agent tries to break what builder creates
7. **Continuous Learning**: Post-mortems feed improvements back into CLAUDE.md
